# SQLite Migration Design

## Goal

Migrate Aggr Hub from PostgreSQL to SQLite to reduce local CPU and memory usage. The app should normally run without a database server process. Existing PostgreSQL data must be copied into SQLite once during migration.

## Selected Approach

Use SQLite as the only runtime database.

- Replace the PostgreSQL Drizzle driver and schema primitives with SQLite equivalents.
- Store the database in a single SQLite file.
- Remove the PostgreSQL service from Docker Compose.
- Add a one-time migration command that reads existing PostgreSQL data and writes it to SQLite.
- Avoid dual PostgreSQL/SQLite runtime support.

This keeps the runtime simple and matches the local resource goal. PostgreSQL compatibility paths are intentionally out of scope.

## Runtime Database

The server will use Drizzle with a SQLite driver. `better-sqlite3` is the preferred driver because it is small, synchronous, and well supported by Drizzle for Node server use.

The SQLite file path will come from `SQLITE_DB_PATH`, with a local default that works for development. Docker Compose will mount a data volume and set `SQLITE_DB_PATH` to a path under that volume.

The server startup path should not depend on PostgreSQL. Normal development and Docker usage should only require the SQLite file.

## Schema Mapping

The current logical tables remain:

- `feeds`
- `entries`

PostgreSQL-specific schema features will be mapped directly:

- UUID columns become `text` IDs generated by `crypto.randomUUID()`.
- Boolean columns become SQLite integer booleans through Drizzle's boolean mode.
- Timestamp columns become SQLite integer timestamps through Drizzle's timestamp mode.
- PostgreSQL enums become text columns typed in TypeScript.
- Unique constraints remain on `feeds.url` and `(entries.feed_id, entries.guid)`.
- `entries.feed_id` keeps a cascade foreign key to `feeds.id`.

The existing PostgreSQL migration files can remain in git as history, but the active Drizzle config should point to SQLite migrations for future development.

## Query Changes

Most existing Drizzle queries can stay structurally the same. Required SQL changes:

- Replace PostgreSQL tuple cursor comparison with SQLite-compatible conditions.
- Replace the scheduler interval expression with app-side date calculation.
- Preserve `.returning()` only where the selected SQLite driver and Drizzle support it; otherwise use a follow-up select where needed.

Cursor pagination order stays `publishedAt DESC, id DESC`.

## Data Migration

Add a one-time script under the server package.

Inputs:

- `DATABASE_URL`: source PostgreSQL connection string.
- `SQLITE_DB_PATH`: destination SQLite file path.

Behavior:

- Open PostgreSQL as the source.
- Open SQLite as the destination.
- Ensure SQLite migrations have already been applied.
- Copy `feeds` first, then `entries`.
- Preserve IDs, URLs, titles, read/favorite/read-later state, cache headers, summaries, status fields, and timestamps.
- Insert with conflict handling so rerunning the migration does not duplicate rows.

The script does not keep the two databases in sync. It is an import tool, not a replication layer.

## Docker Compose

Remove the PostgreSQL service and server dependency on it.

The server service will:

- Set `SQLITE_DB_PATH=/data/aggr-hub.sqlite`.
- Mount a named volume at `/data`.
- Keep existing OpenAI/Ollama environment variables.

The web service remains unchanged.

## Documentation

Update README and `.env.example` to describe SQLite as the database and document the one-time migration flow from PostgreSQL.

## Testing

Verification should include:

- `vp install` after dependency changes.
- `vp check`.
- `vp test`.
- A server build.
- A data migration dry run against an available PostgreSQL source if one exists locally.

If a local PostgreSQL source is unavailable during implementation, the migration script should still be type-checked and the limitation should be reported.

## Out of Scope

- Runtime support for both PostgreSQL and SQLite.
- Automatic background synchronization from PostgreSQL to SQLite.
- Multi-user write-heavy tuning beyond simple SQLite pragmas if needed.
- Preserving PostgreSQL migration execution for the runtime app.
